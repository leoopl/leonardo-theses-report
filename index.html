1

O que o estudo mede

Cada tarefa parte de um commit real do GitHub que adiciona uma nova dependência PyPI. Cada modelo Claude recebe o estado do repositório no HEAD do commit e precisa reintroduzir a dependência usando apenas o conhecimento de treino.

A versão escolhida é comparada a duas linhas de base — a versão mais recente no PyPI até a data de corte do modelo e a versão que o desenvolvedor humano de fato escolheu — e checada quanto a vulnerabilidades (OSV API) e instalabilidade. Quando o modelo erra, um único prompt de intervenção pede a atualização, isolando seu efeito causal.

Justificativa de cada etapa

Cada etapa controla uma ameaça específica à validade do experimento.

1

Seleção de projetos preparação

Os filtros — ≥200 estrelas, commit nos últimos 6 meses, ≥100 autores distintos e ≥80% de Python — garantem que a tarefa seja real e representativa, não um projeto morto ou de brinquedo. A data de criação anterior a janeiro de 2025 é o que permite tratar o commit como anterior ao cutoff.

2

Seleção de commits/PRs preparação

Restringir a pacotes PyPI puros (sem git URL, caminho local ou pacote só-Conda) torna a defasagem mensurável pelas APIs do PyPI e do OSV. Exigir PR mesclado ou commit no branch padrão fornece a “verdade de campo”: a versão que o humano de fato escolheu.

3

Ambiente Docker preparação

É o coração do controle. Bloquear internet, WebFetch/WebSearch e pip install força o modelo a responder só com conhecimento de treino — exatamente a variável de interesse. Com acesso à rede, o modelo consulta o PyPI e praticamente nunca erra: isso mede a ferramenta, não o modelo. Imagem fixa e auto-update desligado garantem reprodutibilidade.

4

Prompt inicial medição

Replica o commit com instruções idênticas para todos os modelos e exige pin exato (pacote==1.2.3). O pin é necessário: um range (>=, ~=) tornaria a versão escolhida ambígua e impossível de comparar contra as baselines.

5

Verificação inicial RQ1 + RQ3

Mede as três perguntas centrais: a versão é a mais recente disponível até o cutoff? É igual à do desenvolvedor? Instala e builda? Essa etapa produz o status que alimenta a decisão.

6

Decisão — defasada ou vulnerável? decisão

Se a primeira resposta já é pass, a tarefa encerra. Caso contrário, segue para o laço de update — e é isso que isola o efeito causal de um único prompt adicional, um desenho que nenhum trabalho anterior havia montado.

7

Prompt de update RQ2

Injeta uma única variável de intervenção na mesma conversa (“atualize para a versão segura mais recente compatível”), e apenas quando o modelo já falhou. Assim, qualquer mudança observada é atribuível à intervenção.

8

Reverificação e classificação RQ2 → final

Reaplica os mesmos checks — mudou de versão? foi para a mais recente? removeu a vulnerabilidade? ainda compila? — e reclassifica o resultado em pass, outdated, outdated_and_vulnerable ou unresolvable_spec.

4

Resultados do piloto

80 de 90 execuções foram concluídas. A mensagem central: os modelos sempre obedecem à crítica, mas quase nunca a resolvem.

Trocaram de versão após o prompt de update — em todos os seis modelos

Taxa de conclusão dos três modelos Opus — o gradiente segue a capacidade

Em 35% dos casos o modelo trocou de versão e continuou defasado (outdated → outdated). O prompt de update foi essencialmente inútil para falhas de segurança. O gradiente de conclusão acompanha a capacidade: Opus 15/15, sonnet-4.6 errou 1, sonnet-4.5 errou 3, e o Haiku foi o modo de falha dominante, errando 6 de 15.
